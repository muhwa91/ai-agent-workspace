# 프로젝트 가이드 (CLAUDE.md)

> 이 파일은 모든 에이전트(직원)가 작업 시 자동으로 참고합니다.
> 새 프로젝트면 아래 `(채워넣기)` 를 채우고, **이관 프로젝트면 tech-lead 가 분석해 채웁니다.**
> 보편 개발 표준: [docs/STANDARDS.md](docs/STANDARDS.md) · 스택별 설정: [stacks/](stacks/).

## ⭐ 공통 운영 규칙 (모든 직원 필수 준수)
1. **한국어로만 답변한다.** 코드·식별자·로그를 제외한 모든 설명·대화는 한국어로 한다.
2. **질문 로그는 자동 저장된다.** `UserPromptSubmit` hook 이 모든 질문을 `logs/prompts.jsonl` 에 기록한다.
3. **세션 기록은 자동 보존된다.** `Stop` hook 이 매 응답 종료 시 전체 대화를 `logs/sessions/<session_id>.jsonl` 로 저장한다.
4. **의미 있는 작업이 끝나면 `docs/progress.html` 을 갱신한다.**
   - 최신 항목을 `<!-- ENTRIES:START -->` 바로 아래에 추가(맨 위가 최신). 상태 배지 `ok`/`wip`/`fail`.
   - **각 항목은 "요청 → 처리" 두 줄로, 비개발자도 알기 쉽게 쉬운 말로 쓴다**(개발 용어 최소화·결과 중심). 요청 = 사용자가 원한 것, 처리 = 무엇을 했고 결과가 어떤지. (개발자만이 아니라 비개발자도 함께 보는 문서)
   - **`data-agent` 필수**: 각 `.entry` 에 담당 직원 키를 넣는다(직원별 탭 필터용). 표준 키 `main`/`tech-lead`/`backend`/`frontend`/`ui-ux`/`debug`/`qa`/`review`/`security`. 교차 작업이면 공백으로 여러 개(예: `data-agent="backend frontend"`).
5. **품질 게이트를 건너뛰지 않는다.** 그 프로젝트 스택의 린터·포매터·타입체커·테스트를 기능과 함께 처리한다.

## 이 팀은 스택-무관(범용)이다
- 직원들은 **특정 언어에 고정되지 않는다.** 프로젝트를 먼저 **분석**해 그 스택(언어·프레임워크·도구)을 파악하고, **기존 컨벤션을 따른다.**
- 스택별 구체 설정(린트·타입·테스트·CI)은 [stacks/](stacks/) 프로파일에 있다. tech-lead 가 감지된 스택에 맞는 프로파일을 적용한다. (예: `stacks/laravel-vue/`, `stacks/python/`)

## 진행 모델 — 신규 vs 이관 (두 갈래)
사용자가 작업을 요청하면 Claude(메인)가 알아서 진행한다. **프로젝트 성격에 따라 흐름이 다르다.**

### A. 신규 개발 (백지에서 구축)
구조·아키텍처를 세우는 단계라 계획·역할 분담이 중요하다.
1. 요구가 모호하면 핵심만 먼저 질문(2~3개), 명확하면 진행.
2. **tech-lead 가 스택 분석 + 아키텍처·작업분해 계획 제시** → 사장 **1회 승인**.
3. 승인되면 역할 협업: (신규 화면·리디자인이면 `ui-ux-designer`가 디자인 명세 →) **`backend`가 API 계약을 `02-계약.md`에 동결 → 그 후 `backend-engineer`(서버/로직) ∥ `frontend-engineer`(UI, 계약·명세 기반) 병렬 구현** → `qa-tester`(테스트) → 품질게이트 → `code-reviewer` → 민감기능 시 `security-auditor`. (상세: 아래 "협업 규약")

### B. 이관 프로젝트 (기존 코드를 이어서 — 증분 개발) — **위임 우선**
구조·코드가 이미 있으니 **빠른 반복이 기본. 풀 파이프라인 불필요.** 단, **분업이 기본값**이다 — 메인이 다 떠안지 말고 도메인 직원에게 적극 위임한다.
1. **메인(Claude) 직접 = 예외(아래만)**: 오타·1~2줄 수정·설정값 변경·질문 답변·코드 읽기/설명. 그 외 **구현은 직접 하지 않는다.**
2. **그 외 모든 구현 = 도메인 직원에게 위임이 기본**:
   - 화면·UI·디자인·스타일 → `frontend-engineer` · API·비즈니스 로직·DB·데이터 처리 → `backend-engineer`
   - **버그·예외·수치 불일치·간헐 결함의 원인 진단** → `debugger`(근본 원인+최소 수정안 특정) → 그 수정 구현은 back/front. "왜 안 되는지 모르겠다/값이 이상하다/가끔 깨진다"는 먼저 debugger.
   - **신규 화면·리디자인**이면 `ui-ux-designer`가 디자인·UX 설계서를 먼저 내고 `frontend-engineer`가 구현(작은 스타일 수정은 frontend 직접).
   - 애매하면 "직접 할까?"가 아니라 **"어느 직원에게 맡길까?"** 로 먼저 판단한다.
3. **다중 도메인(화면 + 처리과정) = 병렬 동시 호출**: `frontend-engineer` + `backend-engineer` 를 **한 메시지에서 동시에** 띄워 각자 도메인을 병행 처리(직렬 호출 금지). **교차라는 이유로 멈춰 묻지 않는다 — 확인 없이 바로 협업으로 진행한다**(02-계약 동결·되돌리기 어려운 작업·보안 민감 등 기존 멈춤 지점은 그대로 유지).
4. **대형·아키텍처 결정**일 때만 `tech-lead` 계획. 보안 민감 시 `security-auditor`, 머지 전 품질 점검 시 `code-reviewer`. **민감 기능·대형 변경의 머지 전 게이트는 `code-reviewer` ∥ `security-auditor` 를 한 메시지에서 병렬 호출**(직렬 금지, 게이트 시간 단축).

### 완료 조건 — 수정 후 테스트 (필수)
- **구현·로직을 수정하면 `qa-tester`(또는 해당 스택 테스트 명령)로 테스트를 실행해 통과해야 '완료'로 본다.** 테스트가 없으면 최소 동작 확인 후 완료. (문서·오타 등 비코드 사소 수정은 예외)

### 공통 — 멈추고 확인하는 지점
계획 승인 / 요구가 정말 모호 / 되돌리기 어려운 작업(마이그레이션 실행·삭제·배포·외부 전송) / 보안 민감 / `code-reviewer` 🔴 반복. **(교차 도메인 작업 자체는 멈춤 지점이 아니다 — 묻지 말고 협업으로 진행.)**
끝나면 결과를 한 번에 보고하고 `docs/progress.html` 갱신.
> 매 단계를 직접 지시하려면 **"단계별로 진행해줘"**.

## 협업 규약 (중복 제거 · 핸드오프 문서) — 모든 직원 공통
> 목표: **같은 일을 두 번 하지 않고**, 다음 직원이 추측 없이 이어받는다.

### 디스패치 공개 (누가 무엇을)
- 작업 지시를 받으면 **착수 전, 어느 직원이 어느 부분을 맡는지 먼저 밝힌다.** (예: "backend=인증 서비스·세션 / frontend=로그인 폼·토큰 저장 / security=점검")
- 교차 도메인이면 "**02-계약 동결 후 frontend∥backend 병렬**"임을 함께 표기. 단일·소형 수정은 단일 직원(또는 "메인 직접") 명시.

### 참조 먼저 · 단일 소유권 (중복 제거)
- **참조 먼저**: 일 시작 전 해당 기능의 `docs/features/<기능>/` 문서를 **먼저 읽는다.** 이미 결정된 스택·아키텍처·계약을 **재분석·재결정하지 않는다.**
- **단일 소유권**: 산출물엔 주인 1명, 나머지는 **참조만**.

| 산출물 | 주인 |
|---|---|
| 스택·아키텍처 분석 | tech-lead |
| API 계약 | backend-engineer |
| 디자인 시스템 (`DESIGN.md`, 루트) | ui-ux-designer |

- 메인은 직원 호출 시 **관련 문서 경로를 프롬프트에 넣어** 추측·중복을 막는다.

### 기능별 핸드오프 문서 (SSOT) — `docs/features/<기능>/`
| 문서 | 주인(작성) | 내용 |
|---|---|---|
| `01-계획.md` | tech-lead | 스택·아키텍처·작업분해·순서 |
| `02-계약.md` | backend + ui-ux-designer | API 엔드포인트·요청/응답·에러코드 + 화면 명세·상태 (**병렬 전 동결**) |
| `03-구현·검증.md` | back·front(구현) / qa·security·reviewer(결과) | 변경 파일·연동·테스트·보안·리뷰 판정 (누적) |

- 비개발자용 `docs/progress.html`("요청→처리" 쉬운 말)과 **역할 분리**: features 문서는 직원 간 **기술 핸드오프**용. **중복 시 features 가 정본, `progress.html` 은 결과 1줄(요약)만** 남긴다(이중 기록 방지).

### 계약 우선(Contract-First) — 효율의 핵심
- `02-계약.md` 가 **동결된 뒤에만** `backend` ∥ `frontend` **병렬** 구현(불일치 방지). *(이전 "백엔드 먼저 후 프론트" 순차 규칙을 대체)*
- 결과가 계약과 다르면 **계약이 정답** — 계약 쪽이 옳고 다른 쪽이 수정한다.
- 통합·1차 회부 판정 = **메인**. 교차 버그는 재현 위치로 회부(응답=backend / 표시=frontend).

### 02-계약 공동 작성 · 변경 관리
- **소유권 분리**: `02-계약.md` 안에서 **API 계약 = backend**, **화면 명세 = ui-ux-designer** 로 섹션을 나눠 각자 소유.
- **동결 전 합의(계약 리뷰)**: 동결 직전 둘이 정합성을 한 번 맞춘다 — 화면이 요구하는 데이터·동작을 API가 주는지. **불일치(범위·역량)는 `tech-lead`가 "API 확장 vs 화면 조정"을 판정** 후 동결.
- **동결 후 변경 관리**(효율 핵심): 계약 변경이 필요하면(security·qa·reviewer 지적 / 사장 범위 변경) 메인이 ① `02-계약.md`에 변경분·이유를 한 줄 기록 → ② **영향받는 직원만 선택 재호출**(영향 없는 쪽은 재작업 금지) → 양쪽 영향 시 병렬 재호출.

### 경량 예외 (문서가 일보다 커지면 안 됨)
- **문서 ON 객관 기준**: "API 계약 변경 OR 신규 화면"이면 작성, 아니면 생략. (파일 수가 아니라 **계약·화면 영향**으로 판단)
- 오타·1~2줄·작은 스타일 수정은 **문서 생략**. 문서는 의미 있는 기능에만, 표·불릿로 **간결히**.
- **이관 프로젝트 부트스트랩**: 기존 코드 전체 소급 문서화 금지 — **건드리는 기능만, 그 시점부터** 시작한다.
