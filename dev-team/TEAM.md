# 개발팀 운영 가이드 (스택-무관 / 범용)

> **사장: 당신** — 요구사항 제시와 최종 승인 담당.
> 아래 직원 에이전트들이 실무를 분담한다. **이 팀은 특정 언어에 고정되지 않고**, 프로젝트 스택을 분석해 그에 맞춰 일한다.

## 조직 구성

| 에이전트 | 역할 | 주 담당 |
|---|---|---|
| `tech-lead` | 기술이사/PM | 스택 분석, 요구 분해, 아키텍처 결정, 직원 조율 |
| `ui-ux-designer` | UI/UX 디자이너 | 디자인 시스템·IA·UX 플로우·접근성 설계서 (구현 상류, 코드 X / UI 있을 때) |
| `backend-engineer` | 백엔드(범용) | API·로직·DB·마이그레이션 (스택 감지·적응) |
| `frontend-engineer` | 프론트엔드(범용) | 컴포넌트·상태·라우팅·API 연동 (프론트 있을 때) |
| `debugger` | 디버깅 | 재현되는 버그·예외·수치 불일치·간헐 결함의 근본 원인 특정 + 최소 수정안 (구현은 back/front) |
| `qa-tester` | QA | 그 스택의 테스트 도구(Pest/pytest/Vitest 등)·커버리지 |
| `code-reviewer` | 코드 리뷰 | 품질·컨벤션·게이트, 머지 게이트 |
| `security-auditor` | 보안 | 취약점 점검, 배포 전 검사 |

## 표준 워크플로우 (새 기능 / 이관 프로젝트)

```
1. 사장이 요구사항 제시
        │
2. tech-lead 가 스택 분석 + 작업 분해 + 아키텍처 결정 → 01-계획.md → 사장 승인
        │
   (신규 화면·리디자인이면) ui-ux-designer 가 디자인 시스템·레이아웃·UX 설계서 → 02-계약.md(화면)
        │
2.5 backend 가 API 계약(엔드포인트·요청/응답·에러코드)을 02-계약.md 에 **동결**
        │
3. 계약 동결 후 **병렬**: backend-engineer (서버/DB)  ∥  frontend-engineer (UI/연동, 계약·명세 기반)
        │   ※ 둘 다 02-계약.md 를 보고 구현 → 불일치 방지 (프론트 있을 때)
        │
4. qa-tester 가 테스트 작성·실행 (그 스택 도구, 커버리지 80%+)
        │
5. 품질 게이트 통과: 그 스택의 린트·포맷·타입체크·테스트·빌드
        │
6. code-reviewer 가 품질 검토 (🔴 있으면 담당자에게 반려)
        │
7. security-auditor 가 보안 점검 (민감 기능/배포 전)
        │
8. CI(GitHub Actions) 통과 + 사장 최종 확인 후 머지 (Conventional Commits)
```
> 스택별 도구·CI 는 [../stacks/](../stacks/) 프로파일, 보편 표준은 [../docs/STANDARDS.md](../docs/STANDARDS.md).

## 사장(당신)을 위한 호출 치트시트
보통은 **그냥 결과를 말하면** 자동 오케스트레이션으로 진행된다("회원가입 만들어줘"). 직원을 명시하려면:
- 기획·설계·스택분석: "tech-lead한테 이 기능 분해 시켜줘"
- 디자인/UX: "ui-ux-designer로 이 화면 디자인·UX 설계해줘"
- 백엔드: "backend-engineer로 게시글 CRUD API 만들어줘"
- 프론트: "frontend-engineer로 게시글 목록 화면 만들어줘"
- 디버깅: "debugger로 이 버그 원인 찾아줘" / "값이 왜 이상한지 진단해줘"
- 테스트: "qa-tester로 방금 API 테스트 작성하고 돌려줘"
- 리뷰: "code-reviewer로 이번 변경 리뷰해줘"
- 보안: "security-auditor로 로그인 기능 보안 점검해줘"

> 큰 기능은 그냥 **"tech-lead한테 맡겨줘"** 하면 알아서 분해 후 위임한다.

## 팀 규칙 (모든 에이전트 공통)
1. 코드 수정 전 **기존 코드·컨벤션을 먼저 읽고** 따른다. 새 패턴·라이브러리 임의 도입 금지.
2. 결과는 **정직하게** 보고한다. 테스트 실패는 실패라고 말한다.
3. 보안·테스트는 "나중에"가 아니라 기능과 함께 처리한다.
4. **품질 게이트(린트·포맷·타입체크·테스트)를 건너뛰지 않는다.** 통과 못 하면 머지하지 않는다.
5. 되돌리기 어렵거나 외부로 나가는 작업(배포, 삭제, 마이그레이션 실행)은 사장 확인 후 진행.
6. **참조 먼저·중복 금지**: 일 시작 전 `docs/features/<기능>/`(01-계획·02-계약) 문서를 먼저 읽고, 이미 결정된 스택·아키텍처·계약을 재분석하지 않는다. 끝나면 자기 산출물을 해당 문서에 남긴다(단일 소유권). 작은 작업은 문서 생략. (상세: 프로젝트 `CLAUDE.md` "협업 규약")

## 위치 / 커스터마이징
- 에이전트 정의: 이 템플릿은 프로젝트 로컬 `.claude/agents/*.md` 에 직원들을 담는다(폴더를 복사하면 따라온다).
- 스택별 설정은 `stacks/<스택>/`, 프로젝트 고유 컨벤션은 프로젝트 루트 `CLAUDE.md` 에 적으면 모든 에이전트가 참고한다.
